# Diretrizes de Qualidade e Testes (Magic Oven)

Estas regras garantem que o CMS e o Site Público mantenham a estabilidade durante novos desenvolvimentos e atualizações antes de realizar o envio (`git push`) para o repositório remoto.

## 1. Verificações Estáticas e Formatação
Antes de qualquer commit, é fundamental garantir que o código obedeça os padrões da equipe.
*   **Linter (ESLint):** Sempre execute análise estática no código JS (front-end e back-end) para pegar variáveis não utilizadas, falta de *imports* ou erros de sintaxe (como brackets esquecidos).
*   **Formatter (Prettier):** Certifique-se de que o código CSS, HTML e JS esteja formatado utilizando as configurações padrão (indentação, aspas simples/vazias padrão do projeto).

## 2. Testes Unitários e de API (Backend)
Quando houver modificações em `server.js` ou na lógica de banco de dados (`data/`):
*   Deve-se testar as respostas dos endpoints (ex: `GET /api/blog`, `POST /api/auth/login`).
*   Validar estritamente o retorno de códigos de status (ex: 200 OK, 401 Unauthorized, 404 Not Found) em simulações isoladas.

## 3. Integração de Interface (DOM e Javascript Inicial)
Ao modificar utilitários globais do Front-end (ex: `js/cms.js` ou scripts na página `/admin`):
*   Certifique-se de validar se a manipulação do DOM principal está ocorrendo e não lançando erros de console bloqueantes e que as requisições assíncronas do cliente estejam conseguindo consumir o Backend (Integração Lógica).

## 4. Testes End-to-End (E2E)
A experiência do usuário precisa estar intacta nos fluxos essenciais:
*   **Fluxo do Administrador:** O processo de Entrar (Login), Recuperar Senha, Navegar pelas Abas, Criar Conteúdo, Editar Perfis e Excluir itens (modais), deve ser simulado no Browser.
*   **Fluxo do Visitante:** A Home (`index.html`), transições 3D (ASCII Viewer), e páginas como Blog e Portfólio, devem renderizar os componentes carregados dinamicamente sem congelamentos de animação ou falha em carregamento de imagens.

> [!IMPORTANT]
> **Automação (Futuro):** Sempre que essas ferramentas estiverem configuradas nos pacotes e scripts do `package.json` (como Jest e Cypress), o comando primário para rodar essa suíte localmente deve ser preferencialmente acionado antes de cada versão (release). Caso ainda não estejam atrelados aos scripts de automação, valide-os manualmente seguindo este roteiro de forma estrita!
